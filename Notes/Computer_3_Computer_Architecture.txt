## 🔍 DETAILED FUNCTIONAL UNITS EXPLANATION

### 1. INPUT UNIT
**Purpose:** Interface between user and computer

**Functions in Detail:**
- **Data Acquisition:** Captures data from external sources
- **Conversion:** Converts analog signals to digital (ADC - Analog to Digital Converter)
- **Buffering:** Temporarily stores input data
- **Error Detection:** Parity checking, validation

**Common Input Devices with Working Principles:**
| Device | Working Principle | Data Form |
|--------|-------------------|------------|
| Keyboard | Mechanical switches/Matrix | ASCII codes |
| Mouse | Optical sensor/Movement tracking | Coordinates (X,Y) |
| Scanner | CCD/CMOS sensors | Bitmap images |
| Microphone | Diaphragm vibration → Electrical | Digital audio |
| Barcode Reader | Light reflection | Product codes |
| Touch Screen | Capacitive/Resistive sensing | Touch coordinates |

---

### 2. OUTPUT UNIT
**Purpose:** Present processed data to user

**Functions in Detail:**
- **Conversion:** Digital to human-readable form
- **Display Management:** Resolution, color depth
- **Synchronization:** Timing with CPU
- **Buffering:** Store output data before display

**Common Output Devices:**
| Device | Technology | Output Form |
|--------|------------|-------------|
| Monitor | LCD, LED, OLED | Visual display |
| Printer | Laser, Inkjet, Dot Matrix | Hard copy |
| Speaker | Electromagnetic | Sound waves |
| Projector | DLP, LCD | Large display |
| Plotter | Pen/Ink | Vector graphics |

# 2️⃣ CPU REGISTERS - COMPLETE DETAIL

## 🔢 Essential Registers and Their Functions

| Register | Full Form | Size | Purpose |
|----------|-----------|------|---------|
| **PC** | Program Counter | Address width | Holds address of next instruction |
| **IR** | Instruction Register | Instruction width | Holds current instruction |
| **MAR** | Memory Address Register | Address width | Holds memory address for access |
| **MBR/MDR** | Memory Buffer/Data Register | Word size | Holds data to/from memory |
| **AC/ACC** | Accumulator | Word size | Stores intermediate results |
| **SP** | Stack Pointer | Address width | Points to top of stack |
| **BP** | Base Pointer | Address width | Points to stack base frame |
| **FLAGS/PSW** | Program Status Word | 16/32 bits | Stores CPU status flags |
| **GPRs** | General Purpose Registers | Word size | Temporary storage |

**📌 IMPORTANT TRICK:** Remember the sequence - **PC MAR MBR IR**
- PC → MAR (sends address)
- MAR → Memory (selects location)
- Memory → MBR (returns data)
- MBR → IR (instruction loaded)

### Technical Details

| Parameter | Value |
|-----------|-------|
| **CPU Utilization** | 100% during I/O operation |
| **Data Path** | I/O Device → CPU → Memory |
| **Hardware Required** | Status register in I/O device |
| **Software Complexity** | Very Low |
| **Best For** | Simple devices, polling intervals |

### Code Level Representation
```
Programmed I/O (Assembly Level):
    
    CHECK:  IN  STATUS_PORT    ; Read status register
            AND READY_MASK     ; Check ready bit
            JZ  CHECK          ; If not ready, loop
            IN  DATA_PORT      ; Read data
            MOV [MEMORY], AL   ; Store in memory
            ; Continue...
```

### Advantages & Disadvantages

| ✅ Advantages | ❌ Disadvantages |
|--------------|-----------------|
| Simple to implement | Wastes CPU time |
| No special hardware needed | Cannot do parallel work |
| Predictable timing | Slow for fast devices |
| Easy debugging | Inefficient for multiple devices |

## 3.2️⃣ INTERRUPT-DRIVEN I/O - COMPLETE DETAIL

### Interrupt Processing - Step by Step

```
┌─────────────────────────────────────────────────────────────────┐
│              INTERRUPT PROCESSING SEQUENCE                       │
├─────────────────────────────────────────────────────────────────┤
│                                                                   │
│  PHASE 1: INTERRUPT REQUEST                                      │
│  ┌───────────────────────────────────────────────────────────┐  │
│  │ 1. I/O Device completes operation                         │  │
│  │ 2. Device sets interrupt request line HIGH                │  │
│  │ 3. Interrupt Controller (PIC) receives signal             │  │
│  │ 4. PIC prioritizes and forwards to CPU                    │  │
│  └───────────────────────────────────────────────────────────┘  │
│                                                                   │
│  PHASE 2: CPU RESPONSE                                           │
│  ┌───────────────────────────────────────────────────────────┐  │
│  │ 1. CPU checks interrupt flag (IF)                         │  │
│  │ 2. Completes current instruction                          │  │
│  │ 3. Sends INTA (Interrupt Acknowledge)                     │  │
│  │ 4. Disables further interrupts (optional)                 │  │
│  └───────────────────────────────────────────────────────────┘  │
│                                                                   │
│  PHASE 3: CONTEXT SAVE                                           │
│  ┌───────────────────────────────────────────────────────────┐  │
│  │ 1. Push Program Counter (PC) onto stack                   │  │
│  │ 2. Push Processor Status Word (PSW)                       │  │
│  │ 3. Push general purpose registers (if needed)             │  │
│  │ 4. Save current stack pointer                             │  │
│  └───────────────────────────────────────────────────────────┘  │
│                                                                   │
│  PHASE 4: INTERRUPT VECTOR FETCH                                 │
│  ┌───────────────────────────────────────────────────────────┐  │
│  │ 1. CPU reads interrupt vector number from device          │  │
│  │ 2. Uses vector as index into Interrupt Vector Table       │  │
│  │ 3. Fetches ISR address from IVT                           │  │
│  │ 4. Loads ISR address into PC                              │  │
│  └───────────────────────────────────────────────────────────┘  │
│                                                                   │
│  PHASE 5: INTERRUPT SERVICE ROUTINE (ISR) EXECUTION              │
│  ┌───────────────────────────────────────────────────────────┐  │
│  │ 1. Execute ISR code                                       │  │
│  │ 2. Transfer data if needed                                │  │
│  │ 3. Send EOI (End of Interrupt) to PIC                     │  │
│  │ 4. Restore context (pop registers)                        │  │
│  └───────────────────────────────────────────────────────────┘  │
│                                                                   │
│  PHASE 6: RESUME                                                 │
│  ┌───────────────────────────────────────────────────────────┐  │
│  │ 1. Restore Program Counter from stack                     │  │
│  │ 2. Restore PSW from stack                                 │  │
│  │ 3. Re-enable interrupts (if disabled)                     │  │
│  │ 4. Resume original program                                │  │
│  └───────────────────────────────────────────────────────────┘  │
│                                                                   │
└─────────────────────────────────────────────────────────────────┘
```

### DMA Operation Modes

| Mode | Description | Use Case |
|------|-------------|----------|
| **Single Transfer Mode** | Transfers 1 byte/word per request | Slow devices |
| **Block Transfer Mode** | Transfers entire block continuously | Fast devices, disk |
| **Demand Transfer Mode** | Transfers until device deasserts request | Variable rate devices |
| **Cascade Mode** | Multiple DMACs connected | Complex systems |

### DMA Transfer Process - Step by Step

```
┌─────────────────────────────────────────────────────────────────┐
│                   DMA TRANSFER SEQUENCE                          │
├─────────────────────────────────────────────────────────────────┤
│                                                                   │
│  STEP 1: DMA Initialization (CPU Programs DMAC)                  │
│  ┌───────────────────────────────────────────────────────────┐  │
│  │ CPU writes to DMAC registers:                             │  │
│  │ • Source Address (Where data comes from)                  │  │
│  │ • Destination Address (Where data goes to)                │  │
│  │ • Transfer Count (How many bytes)                         │  │
│  │ • Transfer Mode (Single/Block/Demand)                     │  │
│  │ • Transfer Type (Read/Write/Verify)                       │  │
│  └───────────────────────────────────────────────────────────┘  │
│                                                                   │
│  STEP 2: I/O Device Request                                      │
│  ┌───────────────────────────────────────────────────────────┐  │
│  │ I/O device ready for transfer                             │  │
│  │ Device asserts DREQ (DMA Request) to DMAC                 │  │
│  └───────────────────────────────────────────────────────────┘  │
│                                                                   │
│  STEP 3: DMAC Requests Bus Control                               │
│  ┌───────────────────────────────────────────────────────────┐  │
│  │ DMAC asserts HRQ (Hold Request) to CPU                    │  │
│  │ CPU completes current bus cycle                           │  │
│  │ CPU floats its buses (high impedance state)               │  │
│  │ CPU asserts HLDA (Hold Acknowledge) to DMAC               │  │
│  └───────────────────────────────────────────────────────────┘  │
│                                                                   │
│  STEP 4: DMAC Controls Bus                                       │
│  ┌───────────────────────────────────────────────────────────┐  │
│  │ DMAC takes control of address, data, control buses        │  │
│  │ DMAC asserts DACK to I/O device                           │  │
│  │ DMAC places address on address bus                        │  │
│  │ DMAC generates control signals (MEMR, MEMW, IOR, IOW)     │  │
│  └───────────────────────────────────────────────────────────┘  │
│                                                                   │
│  STEP 5: Data Transfer                                           │
│  ┌───────────────────────────────────────────────────────────┐  │
│  │ For I/O → Memory:                                          │  │
│  │   • IOR (I/O Read) active                                 │  │
│  │   • MEMW (Memory Write) active                            │  │
│  │   • Data flows from I/O to Memory via data bus            │  │
│  │                                                           │  │
│  │ For Memory → I/O:                                          │  │
│  │   • MEMR (Memory Read) active                             │  │
│  │   • IOW (I/O Write) active                                │  │
│  │   • Data flows from Memory to I/O via data bus            │  │
│  └───────────────────────────────────────────────────────────┘  │
│                                                                   │
│  STEP 6: Repeat or Finish                                        │
│  ┌───────────────────────────────────────────────────────────┐  │
│  │ Decrement word count                                       │  │
│  │ Increment addresses                                        │  │
│  │ If count > 0 and mode = block, repeat STEP 5              │  │
│  │ If count = 0, DMAC deasserts HRQ                          │  │
│  │ CPU regains bus control                                    │  │
│  │ DMAC may interrupt CPU on completion                       │  │
│  └───────────────────────────────────────────────────────────┘  │
│                                                                   │
└─────────────────────────────────────────────────────────────────┘
```

### Bus Acquisition Methods in DMA

| Method | Description | Advantage | Disadvantage |
|--------|-------------|-----------|--------------|
| **Burst Mode** | DMAC holds bus for entire transfer | Fast transfer | CPU starved |
| **Cycle Stealing** | One word per bus cycle | CPU gets bus frequently | Slower transfer |
| **Transparent Mode** | DMA when CPU not using bus | No CPU slowdown | Complex timing |

## 4.1️⃣ INTERRUPT TYPES - DETAILED EXPLANATION

### A. Hardware Interrupts
Generated by external hardware devices

| Type | Source | Characteristics | Example |
|------|--------|-----------------|---------|
| **I/O Interrupt** | I/O devices | Device ready/completion | Keyboard press |
| **Timer Interrupt** | Programmable timer | Periodic, time slicing | OS scheduler |
| **Power Fail** | Power supply | Emergency, NMI | UPS signal |
| **DMA Interrupt** | DMA controller | Transfer complete | Disk read done |
| **Clock Interrupt** | System clock | Time keeping | Real-time clock |

### B. Software Interrupts
Generated by programs intentionally

| Type | Instruction | Use |
|------|-------------|-----|
| **System Calls** | INT 21h (DOS), INT 80h (Linux) | OS services |
| **Breakpoint** | INT 3 (x86) | Debugging |
| **Overflow** | INTO (x86) | Check overflow |
| **Bound** | BOUND (x86) | Array bounds check |

### C. Exceptions (Internal Interrupts)
Generated by CPU due to abnormal conditions

| Exception | Cause | Handling |
|-----------|-------|----------|
| **Faults** | Correctable error | Restart instruction |
| **Traps** | Deliberate | Continue next instruction |
| **Aborts** | Unrecoverable error | Terminate program |

| **Fixed Priority** | Pre-assigned priority levels | Simple | Rigid |
| **Round Robin** | Equal priority, cyclic | Fair | Complex |
| **Daisy Chain** | Priority by physical position | Easy wiring | Slow for far devices |
| **Programmable** | Priority set by software | Flexible | Needs PIC |

# 5️⃣ ARCHITECTURE COMPARISONS - COMPLETE

## 5.1️⃣ VON NEUMANN vs HARVARD - EXTENDED

### Von Neumann Architecture
```
┌─────────────────────────────────────────────────────────────────┐
│                    VON NEUMANN ARCHITECTURE                      │
├─────────────────────────────────────────────────────────────────┤
│                                                                   │
│  CPU ──────────────────────────────────────┐                    │
│   │                                        │                    │
│   │      ┌─────────────────────────┐       │                    │
│   │      │    SINGLE MEMORY        │       │                    │
│   │      │  ┌─────────┬─────────┐  │       │                    │
│   │      │  │  DATA   │ INSTR   │  │       │                    │
│   └──────│─▶│ SECTION │ SECTION │◀─│───────┘                    │
│          │  └─────────┴─────────┘  │                            │
│          │  (Programs + Data)       │                            │
│          └─────────────────────────┘                            │
│                                                                   │
│  Key Features:                                                   │
│  • Stored Program Concept (Instructions = Data)                 │
│  • Single Bus for both Data and Instructions                    │
│  • Sequential Execution                                          │
│                                                                   │
│  Limitations:                                                    │
│  • Von Neumann Bottleneck (Bus contention)                      │
│  • Cannot fetch instruction and data simultaneously             │
│                                                                   │
└─────────────────────────────────────────────────────────────────┘
```

### Harvard Architecture
```
┌─────────────────────────────────────────────────────────────────┐
│                     HARVARD ARCHITECTURE                         │
├─────────────────────────────────────────────────────────────────┤
│                                                                   │
│  CPU ───────────────────────────────────────────────────────┐   │
│   │                      │                                  │   │
│   │      ┌─────────────┐ │      ┌─────────────┐           │   │
│   │      │ DATA MEMORY │ │      │ INSTR MEMORY│           │   │
│   │      │ ┌─────────┐ │ │      │ ┌─────────┐ │           │   │
│   └──────│▶│  DATA   │◀│─┘      │ │ INSTR   │◀│───────┘   │   │
│          │ └─────────┘ │        │ └─────────┘ │           │   │
│          │  (RAM)      │        │  (ROM/Cache)│           │   │
│          └─────────────┘        └─────────────┘           │   │
│                │                       │                   │   │
│                ▼                       ▼                   │   │
│          ┌─────────────┐        ┌─────────────┐           │   │
│          │  DATA BUS   │        │ INSTR BUS   │           │   │
│          └─────────────┘        └─────────────┘           │   │
│                                                              │   │
│  Key Features:                                                │
│  • Separate Memory for Data and Instructions                 │
│  • Separate Buses for Data and Instructions                  │
│  • Parallel Access possible                                  │
│                                                              │
│  Advantages:                                                 │
│  • Faster (Parallel fetch)                                   │
│  • No bottleneck                                             │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

### Complete Comparison Table

| Feature | Von Neumann | Harvard |
|---------|-------------|---------|
| **Memory** | Single memory space | Separate data & instruction memory |
| **Buses** | Single bus for both | Separate data & instruction buses |
| **Speed** | Slower (bottleneck) | Faster (parallel access) |
| **Cost** | Lower | Higher |
| **Complexity** | Simple | Complex |
| **Fetch** | Sequential fetch | Parallel fetch possible |
| **Flexibility** | More flexible | Less flexible |
| **Self-modifying code** | Possible | Difficult |
| **Data as Instructions** | Yes | No |
| **Examples** | Most PCs | DSP, Microcontrollers |
| **Bottleneck** | Von Neumann Bottleneck | No bottleneck |
| **Memory Utilization** | Efficient | May be inefficient |
| **Instruction Prefetch** | Limited | Advanced |

**📌 EXAM TRICK:** 
- "Von" sounds like "One" → One memory, One bus
- "Harvard" has "Hard" (Harder to implement, but faster)

## 5.2️⃣ RISC vs CISC ARCHITECTURE

### RISC (Reduced Instruction Set Computer)

| **Examples** | x86, 8086, 68000 |

# 7️⃣ PERFORMANCE METRICS - FORMULAS & TRICKS

## Key Performance Formulas

### 1. Clock Speed
```
Clock Speed = 1 / Clock Period

Example: 
Clock Period = 2 ns → Clock Speed = 1/(2×10⁻⁹) = 500 MHz
```

### 2. CPU Time
```
CPU Time = Instructions × CPI × Clock Cycle Time

Or
CPU Time = Instructions × CPI / Clock Speed
```

### 3. MIPS (Million Instructions Per Second)
```
MIPS = Clock Speed / (CPI × 10⁶)

Example:
Clock = 2 GHz, CPI = 2
MIPS = (2×10⁹) / (2×10⁶) = 1000 MIPS
```

### 4. MFLOPS (Million Floating Point Operations)
```
MFLOPS = Number of FP Operations / (Execution Time × 10⁶)
```

### 5. CPI (Cycles Per Instruction)
```
CPI = Total CPU Cycles / Instruction Count
```

### 6. Throughput
```
Throughput = Instructions / Time
```

### 7. Speedup (Amdahl's Law)
```
Speedup = 1 / [(1 - P) + P/S]

Where:
P = Parallel portion
S = Speedup of parallel portion
```

**📌 PERFORMANCE TRICKS:**
- Higher Clock ≠ Always Faster (Check CPI)
- RISC usually has lower CPI than CISC
- Pipeline increases throughput, not individual instruction speed
- Cache improves effective memory access time

## ⚡ QUICK SOLVING TRICKS

### Trick 1: Memory Calculation
```
If address lines = n → Memory = 2ⁿ

Quick Powers:
2¹⁰ = 1K
2²⁰ = 1M
2³⁰ = 1G
2⁴⁰ = 1T
```

### Trick 2: Identifying I/O Method
```
Keywords in Question:
- "CPU continuously checks" → Programmed I/O
- "Device sends signal" → Interrupt I/O
- "Direct transfer without CPU" → DMA
- "Bulk data transfer" → DMA
```

### Trick 3: Interrupt Priority
```
Higher Priority:
- Internal Errors (Divide by zero) > NMI > Hardware > Software
```

### Trick 4: Architecture Identification
```
- Single memory, single bus → Von Neumann
- Separate memory, separate buses → Harvard
- Many registers, simple instructions → RISC
- Few registers, complex instructions → CISC
```

### Trick 5: Flag Register Bits
```
"Careless Zebras Often Sit Peacefully Alone"
C - Carry
Z - Zero
O - Overflow  
S - Sign
P - Parity
A - Auxiliary
```
